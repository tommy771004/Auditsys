# 任務控制台 — 對照 Harness Engineering 網站缺失功能分析

> 對照來源：[Harness Engineering Report](https://harness-engineering-report-28208.netlify.app/)
> 
> 核心公式：**Agent = Model + Harness**

---

## 整體概覽

網站描述了一套 **AI Agent 運行環境設計** 的完整架構，核心論點是：當 AI 模型逐漸商品化，工程的核心競爭力從「Prompt Engineering → Context Engineering → **Harness Engineering**」演進。Harness 用「確定性外殼」包裹「非確定性核心（模型）」，使 Agent 在生產環境中可靠運行。

以下逐一比對任務控制台現有功能與網站所述內容的差距。

---

## 一、運作流程（THE LOOP）— 缺失項目

網站描述的閉環流程：`人類意圖 → Harness 鷹架 → 模型推理 → 行動(工具呼叫) → 沙箱+感測器 → 回饋迴圈`

| 網站描述功能 | 目前狀態 | 差距說明 |
|---|---|---|
| **回饋迴圈 (Feedback Loop)** — 測試/Lint/截圖未過 → 自動修正重試 | ❌ 未實作 | 任務失敗後無自動重試機制，需人工重新處理 |
| **自動化驗證閘門** — 通過才允許提交 PR | ❌ 未實作 | 無品質閘門(Quality Gate)機制，任務完成無需經過自動驗證 |
| **閉環自動修正** — 感測器回饋 → 模型自動修正 | ❌ 未實作 | 缺乏「失敗 → 自動修正 → 重新驗證」的完整迴圈 |
| **Two-Retry 策略**（Stripe 案例）— Agent 修不好 2 次就強制人工介入 | ❌ 未實作 | 無重試次數上限與自動升級機制 |
| **Pivot 機制** — 評估失敗時觸發回滾與策略調整 | ❌ 未實作 | 任務失敗後無法自動回滾並調整執行策略 |

## 二、七大元件逐項比對

### 1️⃣ 執行沙箱 (Execution Sandbox) — 「安全網」

> 網站描述：Agent 運行的物理/邏輯隔離環境。控制「權限半徑」— 檔案系統、網路、進程、資料庫存取。使用容器化/VM 確保 Agent 操作不會損害宿主系統。提供回滾能力作為「保險絲」保證。

| 網站描述功能 | 目前狀態 | 差距說明 |
|---|---|---|
| 物理/邏輯隔離環境 | ❌ 未實作 | 任務在同一環境中執行，無沙箱隔離 |
| 權限半徑控制 (檔案/網路/進程/DB) | ❌ 未實作 | 無細粒度權限控制，任務可存取所有資源 |
| 容器化/VM 整合 | ❌ 未實作 | 無 Docker/VM 容器化任務執行能力 |
| 回滾能力 (Rollback) | ❌ 未實作 | 無法在任務失敗後回滾到先前狀態 |
| 環境快照 | ❌ 未實作 | 無執行前後的環境快照比對 |

### 2️⃣ 身份與驗證 (Identity & Authentication) — 「數位身分」

> 網站描述：Agent 在外部系統中的「數位身份」。管理 Agent 如何向 API、資料庫、協作工具驗證身份。使用環境變數、金鑰管理系統等。確保所有呼叫符合企業合規與安全權限。

| 網站描述功能 | 目前狀態 | 差距說明 |
|---|---|---|
| Agent 數位身份管理 | ❌ 未實作 | 任務無獨立身份概念 |
| API 驗證管理 | ❌ 未實作 | 無法管理 Agent 對外部 API 的認證憑證 |
| 金鑰管理系統 | ❌ 未實作 | 無安全的金鑰/密鑰管理機制 |
| 環境變數管理 | ❌ 未實作 | 無任務級別的環境變數配置 |
| 合規與安全權限控制 | ❌ 未實作 | 無企業級合規審計和權限控管 |
| 外部系統認證 (DB/API/協作工具) | ❌ 未實作 | 無法管理對外部系統的認證連接 |

### 3️⃣ 記憶與上下文管理 (Memory & Context) — 「工作記憶 + 長期記憶」

> 網站描述：長期狀態存儲與即時上下文視窗管理。解決「跨對話遺忘」和「上下文焦慮」問題。使用上下文壓縮、狀態機（最小真相來源）、Context Reset 機制。防止 Agent 在長時間任務中進入無限迴圈。

| 網站描述功能 | 目前狀態 | 差距說明 |
|---|---|---|
| 長期狀態存儲 | ⚠️ 部分 | 有設定面板保存偏好，但非 Agent 級別的記憶系統 |
| 上下文視窗管理 | ❌ 未實作 | 無 Context Window 管理策略 |
| Context Reset 機制 | ❌ 未實作 | 無法在任務切換點清除上下文並透過結構化文件傳遞狀態 |
| 上下文壓縮 (Summarization) | ❌ 未實作 | 無自動摘要功能來壓縮脈絡 |
| 防止 Context Rot（上下文腐爛） | ❌ 未實作 | 無機制解決對話變長後信噪比下降的問題 |
| 防止 Context Anxiety（上下文焦慮） | ❌ 未實作 | 無機制解決上下文視窗接近滿時的品質下降 |
| 狀態機（最小真相來源） | ❌ 未實作 | 無 SoR (System of Record) 層級的權威狀態管理 |
| RAG 檢索增強 | ❌ 未實作 | 無 RAG 機制，無法從知識庫檢索相關資訊 |
| 動態脈絡注入 | ❌ 未實作 | 任務執行時無法自動注入當前上下文資訊 |
| 跨 Session 記憶 | ⚠️ 部分 | 稽核追蹤有歷史記錄，但缺乏決策脈絡的結構化存儲 |
| Scratchpad 模式 | ❌ 未實作 | 無暫存筆記/思考板功能 |
| 專案記憶文件 (類似 CLAUDE.md) | ❌ 未實作 | 無持久化專案慣例記憶文件 |

### 4️⃣ 工具調用 (Tool Calls / MCP / Skills) — 「手腳」

> 網站描述：Agent 與外部世界互動的標準介面。透過註冊表管理可用技能。對模型輸出進行嚴格的 Schema 驗證以防止非法 API 呼叫。支援漸進式技能披露（Skill Disclosure）— 按需載入而非一次性全部載入。

| 網站描述功能 | 目前狀態 | 差距說明 |
|---|---|---|
| 工具/技能註冊表 (Registry) | ❌ 未實作 | 無工具庫，無法註冊/管理可用工具 |
| Schema 驗證 | ❌ 未實作 | 模型輸出無結構化驗證機制 |
| Skill Disclosure（漸進式技能披露） | ❌ 未實作 | 無按需載入技能的機制（全部或全無） |
| 檔案讀寫工具 | ❌ 未實作 | 任務無法直接操作檔案系統 |
| API 呼叫工具 | ❌ 未實作 | 任務無法觸發外部 API |
| 資料庫查詢工具 | ❌ 未實作 | 任務無法直接查詢資料庫 |
| 瀏覽器操作工具 | ❌ 未實作 | 無自動化瀏覽器操作能力 |
| 工具呼叫記錄 | ❌ 未實作 | 無法追蹤任務使用了哪些工具 |
| Middleware（中間件） | ❌ 未實作 | 無「攔截器」層來防止錯誤操作或無限迴圈 |

### 5️⃣ 編排與控制流 (Orchestration & Control Flow) — 「指揮家 / 神經系統」

> 網站描述：決定執行邏輯 — 單一 Agent 迴圈 vs 多 Agent 協作。管理任務分解、分配、匯總。透過顯式控制邏輯（如 while-loop）驅動模型經歷 規劃 → 執行 → 驗證。Agent 的「神經系統」。

| 網站描述功能 | 目前狀態 | 差距說明 |
|---|---|---|
| 單一 Agent 迴圈 (while-loop) | ❌ 未實作 | 無 Agent 級別的自動迭代迴圈 |
| 多 Agent 協作 | ❌ 未實作 | 無多 Agent 蜂群協作模式 |
| 任務自動分解 | ❌ 未實作 | 無法將大任務自動拆解為子任務 |
| 平行執行 | ❌ 未實作 | 任務僅能逐一執行，無平行處理 |
| 結果匯總 | ❌ 未實作 | 多個子任務的結果無法自動匯整 |
| Router 模式 | ❌ 未實作 | 無法根據任務類型自動路由到不同處理流程 |
| Pipeline 模式 | ❌ 未實作 | 無任務管線串接功能 |
| Swarm 模式 | ❌ 未實作 | 無多 Agent 蜂群協作模式 |
| 規劃 → 執行 → 驗證 流程 | ❌ 未實作 | 無結構化的三階段執行流程 |
| Human-in-the-Loop 檢查點 | ⚠️ 部分 | 有人工操作介面，但缺乏結構化的審批節點 |

### 6️⃣ 成本與治理 (Cost Governance) — 「斷路器」

> 網站描述：防止失控操作的「斷路器」。包含：Token 預算控制、最大執行步數、錯誤重試閾值（Retry Caps）、保險絲/斷路器機制。防止無限迴圈或幻覺驅動的 Token 浪費。

| 網站描述功能 | 目前狀態 | 差距說明 |
|---|---|---|
| Token 預算控制 | ❌ 未實作 | 無 Token 使用量預算追蹤與限制 |
| 最大執行步數限制 | ❌ 未實作 | 任務無最大步數/迭代次數限制 |
| Retry Caps（重試閾值） | ❌ 未實作 | 無重試次數上限機制 |
| 斷路器 (Circuit Breaker) | ❌ 未實作 | 無異常偵測後自動熔斷的機制 |
| 成本追蹤 | ❌ 未實作 | 無 API 呼叫成本的即時追蹤與報告 |
| 防止無限迴圈 | ❌ 未實作 | 無偵測與中斷無限迴圈的保護機制 |
| 防止幻覺驅動的 Token 浪費 | ❌ 未實作 | 無幻覺偵測或品質把關機制 |

### 7️⃣ 可觀測性 (Observability) — 「黑盒子紀錄器」

> 網站描述：記錄與追蹤 Agent 行為的能力。提供詳細的執行追蹤、結構化日誌、故障分析。讓工程師能像調試傳統軟體一樣調試 Agent。提供「隔天早上」的回顧數據：決策路徑追蹤、成本與延遲指標。

| 網站描述功能 | 目前狀態 | 差距說明 |
|---|---|---|
| 詳細執行追蹤 (Execution Trace) | ⚠️ 部分 | 稽核追蹤有事件記錄，但缺乏 Agent 級別的詳細執行步驟追蹤 |
| 結構化日誌 | ⚠️ 部分 | 有基本事件日誌，但非 Agent 操作級的結構化日誌 |
| 故障分析 (Failure Analysis) | ❌ 未實作 | 無自動化故障根因分析能力 |

## 三、護城河 (Moat) — 基於 Hamilton Helmer「七力量」框架

> 網站核心論點：**護城河不在模型，而在 Harness。**

### 第 0 個護城河 — 速度 (Speed)

| 網站描述 | 目前狀態 | 差距說明 |
|---|---|---|
| 演進速度作為先決條件護城河 | ❌ 不適用 | 任務控制台非 AI 產品，無快速迭代演進的機制 |
| 日更交付、透明迭代 | ❌ 未實作 | 無版本快速迭代與透明變更追蹤 |

### 七層護城河

| # | 護城河 | 網站描述 | 目前狀態 |
|---|---|---|---|
| 1 | **流程權力 (Process Power)** | 「清潔工護城河」— 可靠性、合規、故障恢復的卓越能力 | ❌ 無自動化可靠性/合規/故障恢復流程 |
| 2 | **獨佔資源 (Cornered Resource)** | 專有數據、獨特管線、獨家行業關係 | ❌ 無專有數據積累或獨特處理管線 |
| 3 | **切換成本 (Switching Costs)** | AI Agent 深度嵌入企業工作流 | ❌ 任務控制台未深度整合到業務流程 |
| 4 | **反向定位 (Counter-Positioning)** | 按任務/成果計費而非按人頭 | ❌ 無創新計費模式 |
| 5 | **品牌 (Brand)** | 品牌成為動詞級別的心智佔領 | ❌ 不適用 |
| 6 | **AI 網路效應 (Data Flywheel)** | 使用 → 數據 → 優化 → 更多使用 的飛輪 | ❌ 無數據飛輪機制 |
| 7 | **規模經濟 (Scale Economies)** | 大量客戶攤銷昂貴基礎設施與合規成本 | ❌ 不適用 |

## 八、缺失功能優先建議

> [!IMPORTANT]
> 依影響程度排列，建議優先實作的功能：

### 🔴 P0 — 核心架構缺失（必須建立才有 Harness 概念）
1. **回饋迴圈 + Two-Retry 策略** — 任務失敗後的自動重試與修正，含重試上限
2. **工具註冊與調用框架** — 可擴展的工具 Registry + Schema 驗證 + Middleware
3. **感測器/驗證系統** — 自動化測試、Lint、品質閘門
4. **成本治理（斷路器）** — Token 預算、最大步數、Retry Caps

### 🟡 P1 — 關鍵增強（生產環境所需）
5. **執行沙箱** — 任務隔離執行環境 + 權限半徑控制
6. **身份與驗證** — Agent 身份管理 + 金鑰管理 + 外部系統認證
7. **可觀測性** — 執行追蹤、決策路徑、成本/延遲指標
8. **記憶與上下文管理** — Context Reset、壓縮、防 Context Rot

### 🟢 P2 — 進階功能（競爭力差異化）
9. **編排系統** — 任務分解、Pipeline、Router、多 Agent 協作
10. **Skill Disclosure（漸進式技能披露）** — 按需載入技能
11. **永久環境修復機制** — 從錯誤中學習，建立防護欄
12. **數據飛輪** — 使用→數據→優化 的正向循環

---

> [!CAUTION]
> **本質差異**：任務控制台目前是一個**傳統任務管理工具**（CRUD + 統計 + 稽核），而網站描述的是一套**AI Agent 運行環境設計（Harness Engineering）**。兩者的設計目標與功能層級完全不同。要彌合此差距，需要從架構層級重新設計，而非僅在現有基礎上增加功能。
